# 文化祭引継ぎ書・資料整理設計メモ

## 目的

高校文化祭の引継ぎ書を、次年度担当者が読みやすく、実務で使いやすく、さらにLLMにも読み込ませやすい形で整理する。

前提として、文化祭は企画から運営まで生徒主体で行われる。したがって、引継ぎ書は単なる記録ではなく、次年度が判断・実行・改善できるための運営ナレッジとして設計する。

---

## 1. 基本方針

### 長すぎる引継ぎ書の問題

長い引継ぎ書は情報量が多くても、読む側が全体を把握しにくい。特に文化祭直前や引継ぎ直後は、すべてに目を通す余裕がない可能性が高い。

そのため、今年の引継ぎ書は次の方針で作る。

- 最重要項目を最初に端的に示す
- 詳細は必要になったとき自然に見に行ける構造にする
- 図を多く使い、全体像・連携・判断フローを視覚化する
- 本体にすべて詰め込まず、詳細資料や元資料へ誘導する
- 人間用の引継ぎ書とLLM用ナレッジベースを分けて設計する

---

## 2. 引継ぎ書の3階層設計

引継ぎ書は1つの長いファイルにせず、複数ファイルに分けて3階層で構成する。

```text
第1階層：まず読む
  最重要項目・全体像・読む順番

第2階層：見ながら動く
  時期別マニュアル・業務別マニュアル・他パート連携

第3階層：困ったら見る／詳しく知る
  変更点・反省・トラブル・資料カタログ・元資料
```

### 推奨フォルダ構成

## 3. 第1階層：まず読む資料

第1階層は、次年度担当者が最初に読む資料である。1〜3ページ程度に収める。

### 入れる内容

- このパートの役割
- 最重要5項目
- 今年の大きな変更点
- 来年必ず確認してほしいこと
- 最初に見るべき資料
- この引継ぎ書の読み方

### サンプル

```md
# まず読む：〇〇パート最重要まとめ

## 役割
〇〇パートは、〇〇を安全・円滑に進めるために、各クラス・先生・他パートをつなぐ役割である。

## 最重要
1. 安全に関わることは即確認する。
2. 締切は本当に必要な日の2日前に設定する。
3. 他パートが関係する業務は春休み〜4月中に責任範囲を確認する。
4. 当日の混乱は、前日までの動線確認・備品確認・連絡でかなり防げる。
5. 片付け・ゴミ・返却は準備段階から考える。

## 今年の主な変更
- 〇〇
- 〇〇
- 〇〇

## 来年必ず確認
- 〇〇を継続するか
- 〇〇の担当をどのパートにするか
- 〇〇の資料を修正するか

## 最初に見る資料
1. 年間ロードマップ
2. 他パート連携図
3. 資料カタログ
4. 今年の変更点まとめ
```

## 4. 第2階層：実務マニュアル

第2階層は、実際に運営しながら見る資料である。

### 書き方の型

各時期・各業務について、次の型で統一する。

```md
## 4月：委員会・企画書・場所決定

### この時期の目的
各クラスの企画内容・必要備品・場所を確定させる。

### やること
- 初回委員会を開く
- LINEグループを作る
- 仮企画書を配布・回収する
- 本企画書を配布・回収する
- 場所と列の流し方を決める

### 注意
企画書はその場で確認する。後回しにすると修正連絡が増える。

### 関係する資料
- 仮企画書
- 本企画書
- 備品貸出表
- 注意事項
- 誓約書

### 関係する他パート
- 配置移動
- 会計
- 食品模擬
- 広報・パンフレット
```

### 時期別ロードマップ例

```text
12月
  模擬説明準備・昨年度資料確認
   ↓
1月
  先生顔合わせ・説明日程決定
   ↓
2月
  アンケート集計・企画候補整理
   ↓
3月
  模擬店リスト仮決定・備品調査
   ↓
春休み
  配置・資料・締切・委員会準備
   ↓
4月
  委員会・企画書回収・場所決定
   ↓
5月
  準備状況確認・備品最終調整
   ↓
前日
  備品受け渡し・動線確認・注意連絡
   ↓
1日目
  見回り・混雑対応・閉店後改善
   ↓
2日目
  見回り強化・撤収・返却・会計確認
```

## 5. 第3階層：詳細資料・反省・元資料

第3階層には、長めの反省、判断経緯、詳細なトラブル記録、元資料へのリンクを置く。

### 変更点の書き方

```md
# 今年の変更点：〇〇

## 昨年まで
- どのように運用していたか

## 今年の変更
- 何を変えたか

## 変更理由
- なぜ変えたか
- どんな問題を解決したかったか

## 実施結果
- 良かった点
- 問題になった点
- 想定外だった点

## 来年への提案
- 継続すべきか
- 条件付きで継続か
- やめた方がよいか
- 改善するなら何を変えるか
```

### 反省点の書き方

反省は個人批判ではなく、仕組みの問題として書く。

悪い例：

```text
〇組の委員が全然動かなかった。
```

良い例：

```text
委員への指示がLINEのみで、当日何をすべきかが十分に共有されていなかった。来年は前日までに「当日行動表」を紙または画像で配布するべき。
```

## 6. 図の活用方針

図は、読む負担を減らすために積極的に使う。

### 入れるべき図

1. 全体ロードマップ
2. 他パート連携図
3. 企画決定フロー
4. 資料の使い方マップ
5. 当日対応フロー
6. 撤収フロー
7. 雨天時対応フロー

### 他パート連携図の例

```text
                 実行委員長・副実行委員長
                         │
                         │ 全体判断・雨天・緊急対応
                         │
配置移動 ── 教室模擬 ── 会計
  │             │          │
教室・机椅子     │        価格・売上
暗幕・動線       │        値下げ禁止
                 │
食品模擬 ────── 受付・広報
食品手続き        価格表・案内・パンフ
                 │
              ゴミ・清掃
              撤収・分別
```

### 当日対応フローの例

```text
問題発生
   ↓
安全に関わる？
   ├─ はい → すぐ停止・先生/実長へ
   └─ いいえ
        ↓
他パートが関係する？
   ├─ はい → 該当パートへ共有
   └─ いいえ
        ↓
その場で解決できる？
   ├─ はい → 対応して記録
   └─ いいえ → 閉店後に修正・委員LINEで共有
```

### 資料の使い方マップの例

```text
最初に見る
  ├─ 年間ロードマップ
  ├─ 今年の変更点まとめ
  └─ 資料カタログ

4月に使う
  ├─ 仮企画書
  ├─ 本企画書
  ├─ 備品貸出表
  └─ 注意事項・誓約書

前日・当日に使う
  ├─ 備品返却表
  ├─ 当日チェックリスト
  ├─ 動線図
  └─ LINE連絡文テンプレート

困ったら見る
  ├─ 雨天対応
  ├─ 食品模擬
  ├─ ファストパス
  └─ 片付け・ゴミ処理
```

## 7. 他パート連携ガイド

他パートとの連携は、独立した章または独立ファイルとして作る。

### 基本方針

- 自パートだけで完結する業務は少ない
- 早めに相談すれば解決する問題が多い
- 当日ではなく、春休み〜4月中に責任範囲を確認しておく
- 「誰が最終決定するか」を曖昧にしない

### 各パート説明の型

```md
### 配置移動パート

#### 役割
教室配置、机・椅子・暗幕、廊下や階段の動線、備品移動などを担当するパート。

#### 自パートとの関係
教室模擬では、各クラスの場所決定、列の流し方、机椅子の数、暗幕貸出、前日準備、撤収で強く関係する。

#### 相談すべき時期
- 春休み：教室配置・列形成の方針確認
- 4月：各クラスの企画内容を踏まえた配置調整
- 文化祭前日：机椅子・暗幕・動線の最終確認
- 2日目閉店後：返却・撤収確認

#### 注意点
暗幕や机椅子の管理は教室模擬だけで判断しない。
列が廊下に出る企画は、早めに配置移動と共有する。
```

### 他パート連携表

```md
| 連携先 | 主な役割 | 連携する内容 | 相談時期 | 注意点 |
|---|---|---|---|---|
| 配置移動 | 教室・備品・動線 | 教室配置、机椅子、暗幕、列形成 | 春休み〜前日 | 列が廊下に出る企画は早めに共有 |
| 会計 | 売上・価格管理 | 価格設定、値下げ禁止、会計報告 | 4月〜当日 | 価格変更期限を明確にする |
| 受付 | 来場者案内 | 価格表、混雑案内、場所案内 | 前日〜当日 | 受付に渡す資料を固定する |
| 食品模擬 | 食品管理 | 食品企画、冷蔵庫、衛生管理 | 企画決定前 | 指示系統を一本化する |
| ゴミ・清掃 | ゴミ処理 | 分別、回収場所、撤収 | 前日〜2日目 | 片付け前に必ず確認 |
```

## 8. 資料カタログ

大量の資料を残している場合、本文にすべてを入れず、資料カタログを作る。

### 資料カタログに入れる項目

| 項目 | 内容 |
|---|---|
| 資料名 | ファイル名そのまま |
| 保存場所 | Drive / USB / フォルダ名 |
| 用途 | 何に使う資料か |
| 使用時期 | 12月、春休み、4月、前日、当日など |
| 対象 | 文スタ用、先生用、各クラス用、受付用など |
| 来年使えるか | そのまま使用可 / 要修正 / 参考のみ |
| 修正すべき点 | 日付、クラス数、価格、ルール変更など |
| 関連パート | 会計、配置移動、受付など |
| 注意点 | ミスが起きやすい箇所 |

### 資料カタログ例

```md
| 目的 | 見る資料 | 保存場所 | 使用時期 | 来年の扱い |
|---|---|---|---|---|
| 全体の流れを知る | 年間ロードマップ | Drive/00_最初に読む | 最初 | そのまま使用可 |
| 企画書を配る | 仮企画書・本企画書 | Drive/配布資料 | 4月 | 日付修正 |
| 備品を管理する | 備品貸出表 | Drive/備品 | 4月〜前日 | 備品数確認 |
| 価格を共有する | 価格表 | Drive/価格 | 前日〜当日 | 要方針確認 |
| 当日見回る | 当日チェックリスト | Drive/当日 | 当日 | 使用可 |
| 雨対応を確認する | 雨天対応メモ | Drive/トラブル | 雨予報時 | 参考 |
```

### 資料の優先度表

```md
| 優先度 | 資料 | 理由 |
|---|---|---|
| S | 年間スケジュール、企画書、備品表、当日チェックリスト | これがないと運営できない |
| A | 価格表、配置図、LINE文面、雨天対応 | 混乱防止に必要 |
| B | 会議議事録、写真、過去データ | 判断材料として有用 |
| C | デザイン案、参考資料 | 必要時のみ参照 |
```

## 9. LLM用Markdownの設計

LLM用資料は、人間用の引継ぎ書とは別に作る。

人間用は読みやすさを重視する。LLM用は、検索性・構造・網羅性・出典管理を重視する。

### 推奨構成

```text
99_LLM用/
├─ master.md
├─ index.md
├─ source_index.md
├─ folder_tree.md
├─ glossary.md
├─ unresolved_issues.md
├─ details/
│  ├─ classroom_booths.md
│  ├─ food_booths.md
│  ├─ accounting.md
│  ├─ layout.md
│  └─ cleanup.md
└─ sources/
   ├─ SRC-001_classroom_handover.md
   ├─ SRC-002_budget_sheet.md
   └─ SRC-003_line_messages.md
```

### master.md の構成例

```md
# 文化祭運営ナレッジベース

## 0. この文書の目的
この文書は、文化祭運営に関する資料・引継ぎ・反省・実務知識をLLMに読み込ませるために統合したものである。

## 1. 文化祭全体の概要
- 文化祭名：
- 年度：
- 運営体制：
- 生徒が担当する範囲：
- 教員が関与する範囲：

## 2. パート一覧
### 教室模擬
### 食品模擬
### 配置移動
### 会計
### 受付
### 広報
### 展示

## 3. 年間スケジュール

### source_index.md の構成例

```md
# Source Index

| ID | 資料名 | 種類 | 保存場所 | 内容 | 信頼度 |
|---|---|---|---|---|---|
| SRC-001 | 教室模擬引き継ぎ書2025-2026.pdf | PDF | 30_元資料/引継ぎ | 教室模擬の年間流れ・反省・資料構成 | 高 |

## 10. 大量資料をLLMに読み込ませる方法

大量の資料を一気に渡すのではなく、段階的に処理する。

### 推奨フロー

```text
資料フォルダ
   ↓
フォルダ階層を出力
   ↓
ファイル一覧を作成
   ↓
形式ごとにテキスト抽出
   ↓
各ファイルを短く要約
   ↓
重要資料だけ詳細要約
   ↓
source_index.md 作成
   ↓
master.md 作成
   ↓
人間が確認・修正
   ↓
LLMに渡す
```

### 読み込ませる順番

```text
Step 1：フォルダ階層だけ渡す
↓
Step 2：ファイル一覧＋更新日＋簡単なメモを渡す
↓
Step 3：LLMに重要度分類をさせる
↓
Step 4：重要ファイルから順に中身を渡す
↓
Step 5：各ファイルを要約させる
↓
Step 6：要約を統合して master.md を作る
↓
Step 7：source_index.md を作る
```

### 形式ごとの処理方法

| 形式 | 処理方法 |
|---|---|
| .md | そのまま読み込み |
| .txt | そのまま読み込み |
| .docx | python-docx などで抽出 |
| .pdf | pypdf などで抽出。ただし図表は確認が必要 |
| .xlsx | openpyxl などで抽出 |
| .pptx | python-pptx などで抽出 |
| .csv | pandas などで抽出 |
| 画像 | OCRまたは手動要約。重要画像は人間が確認 |

## 11. フォルダ階層の渡し方

LLMに資料を渡すときは、最初にフォルダ階層だけを渡す。

### 目的

- 全体構造を把握させる
- どの資料が重要そうかを推定させる
- 次に読むべきファイルの優先順位を決める
- 不足資料や重複資料を見つける

### フォルダ階層を渡すときのプロンプト

```md
以下は文化祭引継ぎ資料のフォルダ階層です。

目的：
- 次年度の文化祭スタッフが使いやすい引継ぎ書を作る
- 大量の資料を整理して、LLM用ナレッジベースを作る
- 重要資料、実務資料、参考資料、元資料を分類する

やってほしいこと：
1. このフォルダ構成を読み取り、資料群の全体像を整理してください
2. 各フォルダの役割を推定してください
3. 最初に読むべき資料を選んでください
4. 次に中身を確認すべきファイルの優先順位をつけてください
5. 不足していそうな資料や、追加した方がよい資料を指摘してください
6. LLM用Markdownに統合する場合の構成案を出してください

注意：
- まだファイルの中身は渡していません
- ファイル名と階層だけから判断してください
- 推測と確定情報を分けて書いてください

フォルダ階層：
```text
ここにフォルダ階層を貼る
```
```

### Windowsでフォルダ階層を出力する方法

対象フォルダでPowerShellを開き、以下を実行する。

```powershell
tree /F
```

テキストファイルに保存する場合：

```powershell
tree /F > folder_tree.txt
```

日本語ファイル名が文字化けする場合：

```powershell
chcp 65001
tree /F > folder_tree.txt
```

更新日やサイズも含めたい場合：

```powershell
Get-ChildItem -Recurse | Select-Object FullName, LastWriteTime, Length | Out-File folder_list.txt -Encoding utf8
```

## 13. 最終的な設計方針

今年の引継ぎ書は、次の構造にするのが最適である。

```text
短い本体
  ↓
図解された全体像
  ↓
時期別・業務別マニュアル
  ↓
困ったときの詳細資料
  ↓
資料カタログ
  ↓
LLM用ナレッジベース
```

### 最も重要な考え方

- 最初の1ページだけで最低限動けるようにする
- 図で全体像・連携・判断フローを示す
- 大量の資料は本文に詰め込まず、目的別索引で活かす
- 引継ぎ書本体は「地図」、元資料は「道具」として分ける
- LLM用資料は人間用資料と分け、網羅性と出典管理を重視する
