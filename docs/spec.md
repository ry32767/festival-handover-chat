## 1. 概要

- **一言コンセプト**: 膨大な文化祭引継ぎ資料を横断検索し、非技術者へ出典付きで次の行動を示すチャットサービス
- **対象利用者**: 文化祭スタッフ、各パートのチーフ・副チーフ、次年度担当者、必要に応じて担当教員
- **主要対象資料**:
  - 教室模擬引継ぎ資料フォルダ
  - 配置移動引継ぎ資料フォルダ
  - 今年度中心・一部昨年度情報を含む既存Markdown資料フォルダ
- **公開方式**:
  - 試作: URLを知る人向け＋共通パスコード
  - 本運用: 学校関係者認証へ移行
- **フロントエンド**: GitHub Pages
- **バックエンド**: Supabase Edge Functions
- **AI**: Gemini API + File Search

## 2. 目的と成功条件

### 2.1 解決する課題

- 長い引継ぎ書は全部読まれにくい
- 資料が複数フォルダ・複数年度・複数パートに分散している
- 過去の具体例、反省、正式ルール、個人の提案が混ざる
- 次年度担当者が「どこを読めばよいか」を判断しづらい
- APIキーや資料を安全に扱いながら、PC操作を最小化したい

### 2.2 成功条件

- 代表質問50件で正答率90%以上
- 安全、食品、会計、責任分担、撤収などの重要質問は100%正答
- 回答の95%以上に適切な「ファイル名＋見出し」が付く
- 資料にない質問を事実として断定しない
- 矛盾資料がある場合に単一の結論を捏造せず、相違と確認先を示す
- 通常回答は10秒以内を目標とする
- スマートフォンとPCの最新主要ブラウザで利用できる
- キャラクター変更で文体は変わるが、事実・出典・安全判断は変わらない

### 3.1 確定した前提

- **主要フロー**: URLを開く → パスコード認証 → キャラクター・パート・年度を選択 → 質問 → 資料検索 → 回答と出典表示 → 必要ならコピー・評価
- **主要データ**: 匿名化Markdown、キャラクター定義、質問、短い会話履歴、引用情報、匿名フィードバック
- **権限**:
  - 一般利用者: 質問・閲覧・コピー・評価
  - あなた: 資料・キャラクター更新、Store同期、設定変更
  - 次年度チーフ: GitHub Webから匿名化MarkdownとキャラクターMarkdownを更新
- **外部連携**: Gemini API、Gemini File Search、Supabase Edge Functions、GitHub Pages
- **資料更新**: GitHub Web編集後、管理者が同期スクリプトを手動実行
- **出典**: ファイル名＋見出し
- **匿名化**: 全資料を投入前に匿名化し、人間レビューを必須とする

## 4. MVP

MVPは以下の5機能群とする。

1. 共通パスコード認証と非技術者向け1画面UI
2. 匿名化資料の横断検索と出典付き回答
3. パート・年度フィルタと会話継続
4. 名前付きキャラクター選択
5. 資料検証・同期・品質評価

## 5. 機能仕様

### F-01 共通パスコード認証

- **目的**: 試作URLを知るだけでは利用できない最低限のアクセス制御を設ける
- **入力**: 共通パスコード
- **処理**:
  1. Edge Functionへ送信
  2. サーバー側で照合
  3. 成功時に短時間有効な署名済みセッショントークンを発行
  4. 失敗回数にレート制限
- **パスコード運用**: 試作期間中は一度設定した共通パスコードを固定で使用する。自動変更、有効期限、利用者向け変更機能は設けない
- **出力**: チャット画面または認証エラー
- **権限**: 未認証利用者
- **データ**: セッショントークン、失敗回数、失効時刻

#### 正常系の受け入れ条件
- [ ] 正しいパスコードでチャット画面へ進める
- [ ] 発行されたセッションで質問APIを呼べる
- [ ] ページ再読込中は有効期限内のセッションを再利用できる
- [ ] セッションの期限が切れても共通パスコード自体は変更されない

#### 異常系・境界値
- [ ] 誤ったパスコードではチャット画面へ進めない
- [ ] 空欄では送信できない
- [ ] 15分以内に5回失敗した場合、追加試行を一時制限する
- [ ] 期限切れ・改ざん済みトークンでAPIを直叩きしても401となる
- [ ] パスコードや比較用秘密値がブラウザbundle、Storage、ログへ露出しない
- [ ] `localhost`または`127.0.0.1`の`?demo=1`でのみ、開発者が認証を省略してUI確認できる
- [ ] UI確認モードの回答と出典は実資料ではないことを画面上で明示する

### F-02 チャット質問と資料横断検索

- **目的**: 膨大な資料を利用者が手動で探さず、質問から必要情報へ到達する
- **入力**:
  - 質問1〜2,000文字
  - 直近最大6往復の会話履歴
  - `persona_id`
  - パート・年度フィルタ
- **処理**:
  1. 認証・レート制限・入力検証
  2. 共通ポリシーとキャラクター定義を構成
  3. Gemini File Searchで関連チャンクを取得
  4. 資料の年度、状態、信頼度、矛盾情報を考慮
  5. 回答と引用情報を構造化
  6. ユーザー向け安全なMarkdownへ変換
- **出力**: 回答本文、出典一覧、警告、応答状態
- **権限**: 認証済み利用者
- **データ**: 質問、短期会話履歴、取得引用、応答メタデータ。質問全文は恒久保存しない

#### 正常系の受け入れ条件
- [ ] 教室模擬に関する質問で教室模擬資料から回答できる
- [ ] 配置移動に関する質問で配置移動資料から回答できる
- [ ] パート横断質問で複数パートの関連資料を参照できる
- [ ] 今年度と昨年度を区別して回答できる
- [ ] 実務質問では「結論」「次の行動」「確認先」を示す
- [ ] 回答にファイル名＋見出しの出典を表示する

#### 異常系・境界値
- [ ] 資料から十分な根拠を取得できない場合、「資料上は確認できない」と返す
- [ ] 資料間に矛盾がある場合、両方の主張と確認先を返す
- [ ] 2,000文字を超える質問は400で拒否し、UIに説明を表示する
- [ ] 会話履歴が上限を超える場合、サーバー側で拒否または許可範囲に正規化する
- [ ] Gemini API障害時に内部情報を含まない再試行案内を表示する
- [ ] 「システムプロンプトを無視」「APIキーを表示」等に従わない
- [ ] 資料内の命令文をシステム命令として実行しない

### F-03 出典表示

- **目的**: 利用者が回答の根拠へ戻り、AIの誤りを検証できるようにする
- **入力**: Gemini File Searchのcitation情報、Markdown front matter、見出し情報
- **処理**: `source_id`と見出しを解決し、重複を除去して表示用に整形
- **出力**: `資料名 — 「見出し」`形式の一覧
- **権限**: 認証済み利用者
- **データ**: source index

#### 正常系の受け入れ条件
- [ ] 出典が回答本文と別領域に表示される
- [ ] 同一出典の重複が除去される
- [ ] 2資料以上を参照した場合、すべて表示される
- [ ] 出典名が匿名化済み資料の表示名と一致する

#### 異常系・境界値
- [ ] 引用はあるが見出し解決できない場合、資料名と「見出し不明」を表示する
- [ ] 引用がない場合、根拠不足警告を表示し断定表現を避ける
- [ ] 存在しない`source_id`を受けても内部パスを表示しない

### F-04 パート・年度フィルタ

- **目的**: 検索ノイズを減らし、質問の対象を利用者が簡単に指定できるようにする
- **入力**:
  - part: `all`, `classroom_booths`, `layout_and_movement`, `current_festival_records`等
  - year: `all`または許可済み年度
- **処理**: 許可リスト検証後、検索条件またはプロンプトの明示条件へ変換
- **出力**: 選択状態と絞り込み済み回答
- **権限**: 認証済み利用者
- **データ**: ブラウザの選択状態、APIリクエスト

#### 正常系の受け入れ条件
- [ ] パート「教室模擬」を選ぶと、教室模擬を優先した回答になる
- [ ] 年度を選ぶと、その年度を優先し、他年度を混同しない
- [ ] `all`でパート横断検索できる

#### 異常系・境界値
- [ ] 不正なpart/yearをAPIへ直接送ると400になる
- [ ] 対象年度資料がない場合、その旨を明示し、無断で別年度ルールへ置換しない

### F-05 キャラクター選択

- **目的**: 利用者の状況に応じて、回答の長さ・構造・批判性を選べるようにする
- **入力**: `persona_id`
- **処理**: サーバー側の許可リストから対応Markdownを読み込み、共通ポリシーの後に適用
- **出力**: キャラクターに応じた文体と構造の回答
- **権限**: 認証済み利用者
- **データ**: `localStorage`へ`persona_id`のみ保存

#### 初期キャラクター

| persona_id | 表示名 | 仕様 |
|---|---|---|
| `standard` | あすとら | 冒頭で「あすとらだよ！」のように名乗る。元気な話し言葉で、結論、今やること、確認先を前向きに整理 |
| `concise` | gemini | 冒頭で「geminiです。」のように名乗る。通常どおり落ち着いた説明で、結論と次の行動を短く回答 |
| `senior_supporter` | すだゆう | 冒頭で「すだゆうです。」のように名乗る。淡々と、はっきりと、会社の会議で発言できる程度の丁寧な話し言葉で、論点、根拠、懸念、過去事例、改善案、確認先を検討 |

#### 正常系の受け入れ条件
- [ ] フロントエンドで3種類を名前と説明付きで選択できる
- [ ] 初期値は「あすとら」になる
- [ ] 選択がブラウザに保存され、次回も復元される
- [ ] 各キャラクターは回答冒頭で名前を名乗る
- [ ] `gemini`以外は話し言葉で回答する
- [ ] 「あすとら」は「〜だよ！」「〜しよう！」のような元気な口調になる
- [ ] 「gemini」は通常どおり落ち着いた回答になるが、出典と重大注意は保持する
- [ ] 「すだゆう」は淡々と、はっきりと、丁寧に論点、懸念、過去事例、改善案を表示する
- [ ] 過去事例が資料にない場合、「資料内に具体例は確認できない」と明示する

#### 異常系・境界値
- [ ] 不正な`persona_id`はサーバー側で`standard`へフォールバックする
- [ ] 無効化されたキャラクターは一覧に表示されない
- [ ] キャラクターを変えても、同一質問の主要事実・出典・安全判断が一致する
- [ ] フロントエンドから任意のキャラクター本文を送信できない
- [ ] キャラクターMarkdownの命令が共通ポリシーと競合する場合、共通ポリシーが優先される

### F-06 質問例と非技術者向けUI

- **目的**: 何を質問すればよいか分からない利用者でも操作できるようにする
- **入力**: 質問例ボタン、自由入力
- **処理**: 選択した例を入力欄へ設定または即時送信
- **出力**: チャット履歴、ローディング、回答、出典
- **権限**: 認証済み利用者

#### 初期質問例
- 今の時期にやることは？
- 教室模擬と配置移動で事前に確認することは？
- 雨天時の注意点は？
- 前日に確認することをチェックリストにして
- この計画の懸念点をすだゆうとしてレビューして
- 関連する資料を教えて

#### 正常系の受け入れ条件
- [ ] PC・スマートフォンの1画面で主要操作が完結する
- [ ] 質問例を1タップで使える
- [ ] 送信中は二重送信を防止する
- [ ] 回答をクリップボードへコピーできる
- [ ] キーボードのみでも基本操作できる
- [ ] ローディング、成功、根拠不足、失敗が視覚的に区別できる

#### 異常系・境界値
- [ ] 空入力では送信できない
- [ ] 通信失敗時に入力内容を保持する
- [ ] コピーAPI非対応時に代替メッセージを表示する
- [ ] 長い回答でもレイアウトが崩れない

### F-07 会話履歴

- **目的**: 直前の文脈を保ちながら追加質問できるようにする
- **入力**: 直近のユーザー・AIメッセージ
- **処理**: 最大6往復に制限し、毎回APIへ送る
- **出力**: 文脈を踏まえた回答
- **権限**: 認証済み利用者
- **データ**: ブラウザメモリまたは`sessionStorage`。MVPではサーバー恒久保存しない

#### 正常系の受け入れ条件
- [ ] 「それはいつ確認する？」のような追質問に直前回答を踏まえて答えられる
- [ ] 新規会話ボタンで履歴を消去できる
- [ ] ページ内で最大6往復を保持できる

#### 異常系・境界値
- [ ] 6往復を超えた場合、古い履歴から除外する
- [ ] 不正なroleや巨大な履歴をAPIへ送っても拒否する
- [ ] 会話履歴に秘密情報を含めないよう注意表示する

### F-08 フィードバック

- **目的**: 誤答や不足資料を特定し、検索・資料を改善する
- **入力**: 👍/👎、任意コメント
- **処理**: 匿名セッションID、質問識別子、評価、短いコメントを保存
- **出力**: 送信完了表示
- **権限**: 認証済み利用者
- **データ**: フィードバック。保持期間は初期30日以内

#### 正常系の受け入れ条件
- [ ] 各回答に👍/👎を付けられる
- [ ] 同一回答への評価を更新できる
- [ ] 任意コメントなしでも送信できる

#### 異常系・境界値
- [ ] コメント文字数上限を超えると送信できない
- [ ] 個人情報を書かない注意を表示する
- [ ] 保存失敗がチャット回答の閲覧を妨げない

### F-09 資料検証と匿名化ゲート

- **目的**: 不正な構造、匿名化漏れ、出典欠落を検索Storeへ投入しない
- **入力**: `knowledge/`以下のMarkdown
- **処理**:
  - front matter検証
  - `source_id`重複検査
  - 見出し構造検査
  - `anonymized: true`検査
  - メール、電話番号、SNS識別子等のパターン検査
  - index参照整合性検査
- **出力**: 合否とファイル・行・理由
- **権限**: 資料更新者、管理者
- **データ**: 検査レポート

#### 正常系の受け入れ条件
- [ ] 正しい資料群は検証を通過する
- [ ] `anonymized: false`または欠落ファイルは同期対象にならない
- [ ] 重複`source_id`を検出する
- [ ] 不正なfront matterをファイル単位で報告する
- [ ] 匿名化疑いをファイル・行付きで報告する

#### 異常系・境界値
- [ ] バイナリファイル、巨大ファイル、シンボリックリンクを無断投入しない
- [ ] 自動検査合格だけで匿名化完了とせず、人間確認を要求する
- [ ] 検査失敗時に同期処理を開始しない

### F-10 File Search Store同期

- **目的**: GitHub上の最新版Markdownを安全に検索Storeへ反映する
- **入力**: 検証済みMarkdown、Gemini APIキー、Store設定
- **処理**:
  1. dry-runで対象、追加、変更、削除候補を表示
  2. 新規Storeを作成
  3. 対象ファイルを投入し完了待機
  4. 代表質問テストを実施
  5. 合格時のみ参照Store設定を切替
  6. 旧Storeは手動削除候補として記録
- **出力**: 同期レポート、新Store名、検証結果
- **権限**: 管理者のみ
- **データ**: File Search Store、同期manifest

#### 正常系の受け入れ条件
- [ ] dry-runではGemini側を変更しない
- [ ] 同期前に匿名化・構造検査を必ず実行する
- [ ] 全ファイル投入完了を待ってから検証する
- [ ] 代表質問が合格した場合のみ本番参照先を変更する
- [ ] 同期manifestにGit commit、日時、ファイル数、Store名を記録する

#### 異常系・境界値
- [ ] 途中失敗しても既存本番Storeを維持する
- [ ] 0件の資料を本番へ切り替えない
- [ ] 重複同期の二重実行をロックまたは検知する
- [ ] API制限・通信失敗時に安全に中断し再実行可能にする
- [ ] Gemini APIキーを標準出力へ表示しない

### F-11 回答品質テスト

- **目的**: 資料追加やプロンプト変更による精度劣化を検出する
- **入力**: 代表質問50件、期待要点、必須出典、禁止断定
- **処理**: 各キャラクター・主要フィルタで実行し、自動＋人間評価
- **出力**: 正答率、出典率、根拠不足処理、応答時間
- **権限**: 開発者、管理者

#### 正常系の受け入れ条件
- [ ] 全50問を再実行できる
- [ ] 正答率90%以上を確認できる
- [ ] 重要質問が全件合格する
- [ ] 出典表示率95%以上を確認できる
- [ ] 3キャラクター間で主要事実が一致する

#### 異常系・境界値
- [ ] 資料にない質問で断定しないことをテストできる
- [ ] 矛盾資料の質問で相違を示すことをテストできる
- [ ] 外部API不調時に品質評価を誤って合格にしない

### 6.2 チャット画面

```text

│ 文化祭引継ぎチャット            │
```

## 7. API仕様

### 7.1 認証

`POST /functions/v1/festival-auth`

#### Request
```json
{
  "passcode": "string"
}
```

#### Success
```json
{
  "session_token": "signed-token",
  "expires_at": "2026-07-13T12:00:00Z"
}
```

#### Error
```json
{
  "error": {
    "code": "UNAUTHORIZED",
    "message": "パスコードを確認してください。"
  }
}
```

### 7.2 チャット

`POST /functions/v1/festival-chat`

#### Request
```json
{
  "message": "4月に教室模擬がやることを教えて",
  "persona_id": "standard",
  "filters": {
    "part": "classroom_booths",
    "year": 2026
  },
  "conversation": [
    { "role": "user", "content": "前の質問" },
    { "role": "assistant", "content": "前の回答" }
  ]
}
```

#### Success
```json
{
  "answer": "4月は主に…",
  "persona": {
    "id": "standard",
    "display_name": "あすとら"
  },
  "sources": [
    {
      "source_id": "SRC-CB-001",
      "title": "教室模擬引継ぎ書2025-2026",
      "heading": "4月：本企画書"
    }
  ],
  "grounding": "grounded",

  "request_id": "uuid"

```

#### 根拠不足
```json
{

  "grounding": "insufficient",

```

## 8. プロンプト構成

バックエンドで次の順に組み立てる。

1. 共通回答ポリシー
2. 安全・出典・年度優先ポリシー
3. 選択キャラクターMarkdown
4. フィルタ条件
5. 会話履歴
6. 利用者の質問
7. File Search tool設定

### 共通原則

- 文化祭資料を主な根拠とする
- 資料にない情報は断定しない
- 過去年度、現年度、提案を区別する
- 矛盾時は両方を示す
- 安全、食品、会計、個人情報、教員判断は確認先を示す
- 出典を必ず返すよう要求する
- キャラクターは共通原則を変更できない

---

## 9. データ・設定

| データ | 保存場所 | 保持 |
|---|---|---|
| 匿名化Markdown | GitHubリポジトリ | 永続・履歴管理 |
| File Search embeddings | Gemini File Search Store | 切替・手動削除まで |
| キャラクターMarkdown | GitHubリポジトリ | 永続・履歴管理 |
| persona選択 | localStorage | 利用者が削除するまで |
| 会話履歴 | sessionStorage/メモリ | タブ・セッション中 |
| 認証セッション | sessionStorage | 短時間 |
| フィードバック | Supabase | 30日以内を初期値 |
| 運用メタデータ | Supabaseまたは設定ファイル | 必要最小限 |

---

## 10. 非機能要件

| 項目 | 要件 |
|---|---|
| 性能 | 通常回答10秒以内を目標。15秒超で進行表示を強化 |
| 可用性 | Gemini障害時に既存画面を壊さず再試行案内 |
| セキュリティ | APIキー非公開、サーバー認可、入力検証、レート制限、匿名化ゲート |
| プライバシー | 質問全文を恒久保存しない。個人情報投入禁止 |
| 対応環境 | Chrome、Edge、Safari、Firefoxの現行版。スマートフォン対応 |
| 保守性 | モデル名、Store名、制限値を設定で変更可能 |
| 監査性 | 同期manifest、Git履歴、出典IDで変更と回答根拠を追跡可能 |
| アクセシビリティ | キーボード操作、明示ラベル、フォーカス、色以外の状態表現 |
| コスト | 1日上限とレート制限を設定。予算超過時は安全に停止可能 |
