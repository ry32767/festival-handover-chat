# セキュリティ設計

## 1. 基本方針

GitHub Pagesは静的フロントエンドとしてのみ使う。Gemini APIキー、共通パスコード、File Search Store名などの秘密情報はSupabase Edge FunctionsのSecretに保持し、ブラウザへ配布しない。

## 2. 利用者と権限

### 試作
| 対象 | 閲覧・質問 | 資料更新 | Store同期 | 設定変更 |
|---|---:|---:|---:|---:|
| 共通パスコード利用者 | 可 | 不可 | 不可 | 不可 |
| あなた | 可 | 可 | 可 | 可 |
| 次年度チーフ | 可 | GitHub上で可 | 原則不可 | キャラクター・資料のみ可 |

### 本運用
学校関係者向けGoogleログインまたはSupabase Authへ移行する。試作の共通パスコードは本運用の認証として使用しない。

## 3. 共通パスコード

- 試作期間中は一度設定した共通パスコードを固定で使用し、自動変更、有効期限、利用者向け変更機能は設けない
- セッショントークンの有効期限と共通パスコードの有効期間は別扱いとする
- UI確認用の認証省略は`localhost`または`127.0.0.1`かつ`?demo=1`の場合だけフロントエンドで有効にし、Supabase Edge Functions側の認証は省略しない
- UI確認モードは実資料やGeminiへ接続せず、サンプル回答であることを明示する
- フロントエンドで正解判定しない
- Edge Functionがサーバー側で検証する
- 平文保存しない。Secretには強いハッシュまたは比較用秘密値を保存する
- 成功時は短時間有効の署名済みセッショントークンを返す
- トークンは`sessionStorage`を基本とし、有効期限切れで再認証する
- 失敗回数にレート制限を適用する
- URLやログへパスコードを含めない

## 4. APIキー

- `GEMINI_API_KEY`はSupabase Secretに保存
- GitHub Actionsを使う場合もRepository Secretに保存
- フロントエンドの環境変数へ置かない
- APIキーをログ、エラー、レスポンスへ含めない
- 漏洩時の失効・再発行手順をREADMEまたは運用メモに残す

## 5. API直叩き対策

UIでボタンを隠すだけでは保護にならない。Edge Functionで必ず以下を実施する。

- セッショントークン検証
- CORS Origin許可リスト
- HTTP method検証
- Content-Type検証
- 入力スキーマ検証
- `persona_id`、`part`、`year`の許可リスト検証
- 文字数・会話履歴件数制限
- レート制限
- Gemini応答の構造検証

CORSはブラウザ制御であり、認証の代替ではない。

## 6. 入力制限

| 項目 | 初期制限 |
|---|---:|
| 質問 | 1〜2,000文字 |
| 会話履歴 | 最大6往復 |
| 1メッセージ中のURL | 原則許可するが自動取得しない |
| persona_id | 3つの許可済みIDのみ |
| part | 設定済みenumのみ |
| year | 許可済み年度または`all` |

HTMLをそのまま挿入せず、回答はMarkdownを安全にサニタイズして描画する。

## 7. レート制限

試作初期値:

- 1セッションあたり1分5質問
- 1セッションあたり1日30質問
- パスコード失敗は15分に5回まで
- 上限超過時は429と再試行案内を返す

IPだけに依存せず、署名済みセッションIDと組み合わせる。上限値は設定可能にする。

### 現在の実装状態

パスコード失敗はEdge Functionインスタンス内のメモリで15分間に5回までに制限している。これは単一インスタンス内の試作防御であり、複数リージョン・複数インスタンス間では共有されない。本番公開前に、共有ストアを使う分散レート制限へ置き換える。チャットの「1分5質問・1日30質問」は未実装である。

## 8. 資料の匿名化

`99_LLM用/`は中間成果物であり、そのまま同期しない。`knowledge:import-llm`でメールアドレス、電話番号、LINE識別子、認証情報を検査し、問題のないファイルだけを`knowledge/40_compilations/`へ取り込む。検出ファイルは隔離し、レポートには機密値を出力しない。

実同期は`knowledge:sync -- --execute`で新しいStoreへ投入し、既存Storeを削除・上書きしない。APIキーは`x-goog-api-key`ヘッダーで送信し、URL、標準出力、同期manifestへ含めない。同期後は引用付き代表質問が成功した場合だけ`GEMINI_FILE_SEARCH_STORE`を切り替える。

- `anonymized: true`でない資料は同期スクリプトが拒否
- 電話番号、メール、SNS ID、典型的な個人名表現を自動検査
- 自動検査の合格だけで公開せず、人間レビューを必須とする
- 匿名化前資料は公開リポジトリへ入れない
- 元資料を同一リポジトリへ置く場合はPrivate Repositoryを必須とする

GitHub Pagesの公開元ブランチに`knowledge/30_sources`をそのまま含めない。ビルド成果物へ資料本文が混入していないことを検査する。

## 9. ログとフィードバック

MVPでは質問全文と回答全文を恒久保存しない。

保存可能な情報:
- 匿名セッションID
- 時刻
- 選択パート・年度・persona_id
- 成否、応答時間、エラーコード
- 利用者の👍/👎
- 任意コメントは個人情報注意表示をした上で短期保存

保持期間は初期30日以内とし、本運用前に学校側と確定する。秘密情報、パスコード、APIキー、取得チャンク全文はログへ残さない。

## 10. プロンプトインジェクション対策

- 利用者入力と資料内命令は信頼しないデータとして扱う
- 共通システムポリシーを最優先にする
- 資料内の「以前の指示を無視」等を命令として実行しない
- キャラクター本文はリポジトリ内の許可済みMarkdownだけを使用
- 利用者が任意のpersona本文を送信できない
- 外部URLの内容を自動取得しない
- APIキー、システムプロンプト、内部設定の開示要求には応じない

## 11. エラー処理

ユーザー向けには分類済みメッセージのみ返す。

| code | 表示 |
|---|---|
| `UNAUTHORIZED` | パスコードまたはセッションを確認してください |
| `INVALID_INPUT` | 入力内容を確認してください |
| `RATE_LIMITED` | しばらく待ってから再度お試しください |
| `NO_GROUNDED_ANSWER` | 資料から十分な根拠を確認できませんでした |
| `UPSTREAM_ERROR` | AIサービスとの通信に失敗しました |
| `INTERNAL_ERROR` | 一時的なエラーが発生しました |

内部スタック、Geminiレスポンス全文、Secret値は返さない。

## 12. 本運用への移行条件

- 共通パスコードを廃止
- 学校関係者の認証方式を確定
- 利用規約・プライバシー説明を用意
- ログ保持期間と管理責任者を確定
- 公開リポジトリと非公開資料の境界を監査
- API利用上限・予算アラートを設定
